## Convex backend

- `convex/convex.config.ts` — scaffold. Registers `@robelest/convex-auth` and `@convex-dev/static-hosting` components on the Convex app.
- `convex/schema.ts` — live. Nine tables now: `users`, `guilds`, `guildChannels`, `guildRoles`, `forms`, `submissions`, `auditLog`, `cooldowns`, and `oauthStates`. Guild rows now track both channel and role sync times, cached guild channels drive routing pickers, cached guild roles drive access rules, and submissions now carry `submittedAt` for per day enforcement. Form fields now allow `short`, `paragraph`, `email`, `code`, `select`, `yes_no`, `checkbox`, and `number` types with optional `helperText`, `options`, and per-number `minValue`, `maxValue`, `currencyUnit`. Forms also carry `showModeratorInFooter`, `linkSubmitterOnPublish`, `ticketMode`, and `autoCloseInactiveDays` for support and bounty request flows. Submissions now track `ticketStatus`, `assignedToUserId`, `assignedToUserName`, `assignedAt`, and `lastActivityAt`, indexed via `by_ticketstatus_and_lastactivityat` for the auto-close sweep.
- `convex/http.ts` — live. `auth.http.add(http)` + Discord `/interactions` (Ed25519 verify, PING, role-gated slash command open with classic or Label-wrapped modal depending on field mix, numeric-field hint builder, modal submit persistence with limit handling, custom success messages, MESSAGE_COMPONENT approve/deny button routing with mod role check, deny reason modal open and submit, ticket lifecycle button routing via `ticket:<action>:<submissionId>` custom ids with per-action role gate — `hasAdministratorPermission` parses Discord's Administrator bit, `ticketActionDenial` evaluates Claim (admin or mod or `ticketClaimRoleIds`), Resolve (admin or mod or `ticketResolveRoleIds` or submitter), Close (admin or mod or submitter), Reopen (admin or mod), and Unclaim (admin or mod or current assignee)) + `GET /api/discord/install` (consumes CSRF nonce → exchanges OAuth code → registers guild → redirects back with `?installed` or `?error`) + `registerStaticRoutes`.
- `convex/auth.ts` — scaffold. `createAuth(components.auth, { providers: [OAuth("github", ...)] })` wrapping `arctic` GitHub. Exports `signIn`, `signOut`, `store`. Needs `JWT_PRIVATE_KEY` + `JWKS` from the CLI wizard.
- `convex/staticHosting.ts` — scaffold. Re-exports `generateUploadUrl`, `recordAsset`, `gcOldAssets`, `listAssets`, `getCurrentDeployment`.
- `convex/users.ts` — live. `access` probe query, `me` query, `upsertFromIdentity` mutation, and `lookupBySubject` internal query. All three public functions read email/name/image via `auth.user.viewer(ctx)` because Robel JWTs only carry `sub`. Only `@convex.dev` accounts are admitted; `wayne@convex.dev` is always `owner`, everyone else is `admin`. `lookupBySubject` lets actions reach the Forge-level user row from an auth `subject` without crossing the reactive query layer.
- `convex/lib/access.ts` — live. `isAllowedEmail` and `roleForEmail` pure helpers. Single source of truth for the Forge access list. Owner email reads from the `OWNER_EMAIL` Convex env var with `wayne@convex.dev` as the compile-time fallback so the upstream dev deployment keeps working when the env var is unset.
- `convex/guilds.ts` — live. Guild registry plus routing defaults and role cache reads. Ships `list`, `current`, `listChannels`, `listRoles`, `updateRoutingDefaults`, `disconnect`, `registerFromInstall`, and internal channel and role cache helpers. Also ships `getBotTokenForDelete` (narrow token lookup for the Discord cleanup action) and `logDiscordDeleteSkip` (best-effort audit entry when a Discord DELETE fails after the Convex row is already gone). Strips `botToken` from public responses and keeps guild channel and role choices reactive after refresh.
- `convex/oauthStates.ts` — live. Short-lived CSRF nonce table for the Discord bot install flow. `create` mints a row tied to the admin's user id, `consume` verifies + deletes in one mutation, `gcExpired` sweeps stale rows. Nonces expire after 10 minutes.
- `convex/discord.ts` — live. Discord REST helper actions. `generateInstallUrl` builds an authorize URL with `state=<nonce>` baked in. `exchangeInstallCode` POSTs to `/api/oauth2/token` and returns the `guild` payload Discord attaches when `bot` scope is granted. `refreshGuildChannels` syncs channel metadata, `refreshGuildRoles` syncs role metadata, and `registerCommand` creates or updates a guild slash command for a published form and stores the returned command id. Phase 3 adds `routeSubmission` (scheduled after every insert), `postToModQueue` (posts a review embed with Approve/Deny buttons), `updateModQueueMessage` (PATCHes the embed to show the decision), `publishSubmission` (posts to a text channel or creates a forum thread with templated title and optional applied tag, attaches ticket-mode buttons plus a status-aware footer when the form has ticket mode on, and now calls `buildSubmissionEmbed` with `audience: "destination"` so private field types listed in `isPrivateFieldType` such as email are stripped from the public embed while mod queue and dashboard views keep them), `sendDecisionDM` (opens a DM channel and notifies the submitter), `updatePublishedTicketMessage` (rewrites the published embed footer and button row on every ticket state change), `archiveForumThread` (locks or unlocks a forum thread on ticket close or reopen), `postReplyToSubmission` (posts a bot-authored dashboard reply into the channel or thread with no pings), and `deleteDiscordMessages` (opt-in cleanup scheduled by `submissions.deleteSubmission` that DELETEs the mod queue and destination messages, swallowing 404s). Default permissions remain `328565051456`.
- `convex/forms.ts` — live. Form builder backend. Ships `list`, `get`, `create`, and `update` for the dashboard editor, plus internal publish and interaction helpers (`getForPublish`, `getByCommand`, `getForModalSubmit`, `setPublicationState`). Validates slash command names, field ids, per type field options and helper text, routing selections, cached required, blocked, and moderator role selections, success message size, lifetime and daily submission caps, and now persists the `showModeratorInFooter` preference forwarded to Discord embeds.
- `convex/submissions.ts` — live. Submission backend. `insertFromDiscord` runs `validateAndSanitize` to strip HTML and control characters, check length and email format, verify option ids, and enforce numeric min/max bounds before storing modal submits keyed by guild and form, enforces per user and per day caps, initializes `ticketStatus = open` and `lastActivityAt` when the form has ticket mode on, returns the saved success message, and schedules `internal.discord.routeSubmission` after insert. `routeContext` bundles submission plus form routing plus guild auth for the Discord actions and now carries option metadata, number-field bounds, and ticket lifecycle state so embeds and handlers can render and enforce correctly. `recordDecision` transitions approve or deny, writes an audit row, and schedules publish, mod queue edit, and DM follow-ups. `decide` is the public dashboard counterpart and shares an in-module `applyDecision` helper with `recordDecision` so Approve and Deny buttons on the results page reuse the same audit, publish, and DM pipeline as the Discord buttons without a nested `ctx.runMutation` hop. `recordTicketAction` and the shared `applyTicketAction` helper transition tickets across claim, unclaim, resolve, reopen, and close (and auto_close), write audit rows, and schedule published-message refreshes plus forum thread archive toggles. `sweepAutoCloseTickets` is the hourly cron that closes inactive tickets. `postReply` is the admin dashboard mutation that records an audit row, patches `lastActivityAt`, and schedules `internal.discord.postReplyToSubmission`. `markModQueuePosted`, `markPublished`, and `logRoutingSkip` round out the decision and diagnostics surface. `listForForm` powers the dashboard results page and accepts an `includeHidden` flag. `setHidden` soft-hides or restores a row without touching Discord; `deleteSubmission` hard-deletes the row and can opt in to removing the linked Discord messages via `internal.discord.deleteDiscordMessages`.
- `convex/auditLog.ts` — live. `listForForm` query reads the tail of `auditLog` rows scoped to a form (300 rows, descending) and maps each row into a UI-friendly shape: `severity` of `info` or `error` driven by an allow-list of failure actions (`mod_queue_post_failed`, `publish_text_failed`, etc.), joined submitter name from `submissions`, and `detail` extracted from `metadata.detail`. Metadata is filtered through `sanitizeMetadata` before leaving the server so `body`, `response`, `responseBody`, `stack`, `error`, `rawError`, and `raw` fields never reach the client (2026-04-18 security audit MEDIUM #5). Authenticated via `requireAllowedViewer` so only allow-listed admins can read the log. Audit row writes still happen in `submissions.ts` (`submission_created`, `mod_queue_posted`, `submission_published`, `logRoutingSkip`, ticket actions, dashboard reply) and in `forms.ts` (form lifecycle).
- `convex/lib/discord-types.ts` — planned. Interaction and InteractionType enums shared across http.ts and actions.
- `convex/lib/signature.ts` — planned. Ed25519 verify helper using `discord-interactions`.
- `convex/lib/modal-builder.ts` — planned. Converts a form row into a Discord modal payload.
- `convex/lib/embed-builder.ts` — planned. Renders submission values into a Discord embed.
- `convex/lib/title-template.ts` — planned. Interpolates `{fieldId}` tokens for forum thread titles.
